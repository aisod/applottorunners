01 — Background

What is this permission?

FOREGROUND_SERVICE_LOCATION (added in Android 14 / API 34) is a fine-grained permission that specifically allows a foreground service to access the device's location. Without it, any foreground service that calls GPS APIs will be rejected at runtime on Android 14+.

⚠️

Google Play policy change (August 2024)

Apps targeting API 34+ must declare android:foregroundServiceType="location" on each relevant service AND declare FOREGROUND_SERVICE_LOCATION in the manifest. Failure causes a runtime crash and a Play Store policy rejection.

android/app/src/main/AndroidManifest.xml 
<!-- Internet permission for Supabase -->
<uses-permission android:name="android.permission.INTERNET" />

<<!-- Location (maps, job pickup/drop-off, runner tracking while using the app) -->
<uses-permission android:name="android.permission.ACCESS_FINE_LOCATION" />
<uses-permission android:name="android.permission.ACCESS_COARSE_LOCATION" />

<<!-- Foreground service for Mapbox turn-by-turn navigation -->
<uses-permission android:name="android.permission.FOREGROUND_SERVICE" />
<uses-permission android:name="android.permission.FOREGROUND_SERVICE_LOCATION" />

<!-- ACCESS_BACKGROUND_LOCATION is explicitly REMOVED — app only tracks while visible -->
<uses-permission android:name="android.permission.ACCESS_BACKGROUND_LOCATION"
    tools:node="remove" />
✅

Background location deliberately removed

The manifest explicitly strips ACCESS_BACKGROUND_LOCATION from any merged dependency using tools:node="remove". This confirms the app only tracks location when the user has the app open or the navigation screen visible — no hidden background tracking.

02 — Use Cases

Features that require this permission

Two distinct runtime tasks inside Lotto Runners use foreground location services. Both are runner-facing and only activate after explicit user action.

📡

Live Runner GPS Tracking

When a runner presses "Start Errand", TrackingService.startTracking() opens a continuous GPS stream via Geolocator. Every position update (filtered to 15 m minimum movement) is upserted to the runner_tracking Supabase table so customers see the runner moving in real time on their map.

TrackingService Geolocator Supabase Realtime Runner Dashboard
🗺️

Mapbox Turn-by-Turn Navigation

When a runner taps "Voice Nav" on an errand or transport booking, MapboxNavigationService.startNavigation() launches the flutter_mapbox_navigation full-screen navigator. Mapbox creates its own foreground service (with type location) to keep GPS running while the navigation UI is active, voice prompts fire, and the route recalculates.

📍

Customer Live Tracking View

The CustomerTrackingPage subscribes to Supabase Realtime changes on runner_tracking. Each time the runner moves, the customer's map marker animates to the new coordinates. The customer's own myLocationEnabled flag also uses foreground location to show a "you are here" dot.

03 — Runtime Flow

Step-by-step: how it works

From the moment a runner accepts an errand to GPS data appearing on the customer's screen — here's exactly what happens in code.

1

Runner presses "Start Errand"

In runner_dashboard_page.dart, the _startErrand() handler first updates the errand status in Supabase, then immediately calls TrackingService().startTracking() with the errand ID and runner user ID.

runner_dashboard_page.dart · line 2039–2046
// Mark errand as in_progress in Supabase
await SupabaseConfig.startErrand(errand['id']);

// Begin foreground GPS streaming
await TrackingService().startTracking(
  errand['id'],
  SupabaseConfig.currentUser!.id,
);
2

TrackingService checks permissions

Before opening the GPS stream, TrackingService verifies that location services are enabled and that the user has granted at least LocationPermission.whileInUse. If denied, tracking is aborted gracefully — no crash, no silent failure.

tracking_service.dart · line 26–44
bool serviceEnabled = await Geolocator.isLocationServiceEnabled();
if (!serviceEnabled) { return; }

LocationPermission permission = await Geolocator.checkPermission();
if (permission == LocationPermission.denied) {
  permission = await Geolocator.requestPermission();
  if (permission == LocationPermission.denied) { return; }
}
if (permission == LocationPermission.deniedForever) { return; }
3

GPS stream opens — foreground service active

Geolocator.getPositionStream() uses the platform's foreground location service under the hood (the one guarded by FOREGROUND_SERVICE_LOCATION). Updates are filtered to 15 m distance to balance accuracy and battery life. The initial position is also fetched immediately so the customer sees the runner straight away.

tracking_service.dart · line 51–69
const LocationSettings locationSettings = LocationSettings(
  accuracy: LocationAccuracy.high,
  distanceFilter: 15, // Only update if moved 15 m
);

_positionSubscription =
  Geolocator.getPositionStream(locationSettings: locationSettings)
    .listen((Position position) {
      _updateLocationInDatabase(position); // ← upsert to Supabase
    });
4

Location synced to Supabase in real time

Each GPS event upserts latitude, longitude, heading, and a timestamp into the runner_tracking table keyed by errand_id. A single row per errand is maintained — no unbounded growth.

tracking_service.dart · line 89–98
await SupabaseConfig.client.from('runner_tracking').upsert({
  'errand_id':  _currentErrandId,
  'runner_id':  _currentRunnerId,
  'latitude':   position.latitude,
  'longitude':  position.longitude,
  'heading':    position.heading,
  'updated_at': DateTime.now().toIso8601String(),
});
5

Customer map updates via Supabase Realtime

CustomerTrackingPage subscribes to PostgresChangeEvent.all on the runner_tracking table filtered to the errand ID. When the database row changes, the map marker animates to the new coordinates and the route polyline is recalculated to reflect the runner's current position.

customer_tracking_page.dart · line 102–127
_trackingChannel = SupabaseConfig.client
  .channel('public:runner_tracking:errand_id=eq.$errandId')
  .onPostgresChanges(
    event: PostgresChangeEvent.all,
    table: 'runner_tracking',
    filter: PostgresChangeFilter(
      type:  PostgresChangeFilterType.eq,
      column: 'errand_id',
      value:  errandId,
    ),
    callback: (payload) {
      setState(() {
        _runnerLocation = LatLng(
          payload.newRecord['latitude'],
          payload.newRecord['longitude'],
        );
      });
      _animateMarkerTo(_runnerLocation!); // smooth map animation
      _updateRoutePolyline();            // redraw route to destination
    })
  .subscribe();
6

Optional: Runner taps "Voice Nav" → Mapbox Navigation

When the runner wants turn-by-turn directions, MapboxNavigationService.startNavigation() launches the full Mapbox navigation experience. The flutter_mapbox_navigation SDK internally creates its own foreground service with type location, which is why both FOREGROUND_SERVICE and FOREGROUND_SERVICE_LOCATION must be declared — without them, Android 14+ blocks the service from starting.

runner_dashboard_page.dart · line 1677–1693
// 1. Get runner's current GPS position
final currentPos = await LocationService.getCurrentPosition();

// 2. Geocode the destination address to lat/lng
final coords = await LocationService.getCoordinatesFromAddress(address);

// 3. Launch Mapbox foreground navigation service
await MapboxNavigationService().startNavigation(
  wayPoints: [
    WayPoint(name: 'Start',  latitude: currentPos.latitude,  longitude: currentPos.longitude),
    WayPoint(name: address, latitude: coords['latitude']!, longitude: coords['longitude']!),
  ],
);
04 — Permission Rationale

Why each permission is justified

The table below maps every location-related permission to its specific use case, showing exactly why it is needed and confirming no over-declaration.

Permission Status Required for Justification
ACCESS_FINE_LOCATION Declared GPS tracking, map "my location" button, geocoding current position Runners must share precise street-level location. Customers need their own "you are here" dot.
ACCESS_COARSE_LOCATION Declared Fallback when GPS unavailable, network location Required companion permission alongside FINE; used when satellite lock is slow.
FOREGROUND_SERVICE Declared Running the GPS stream while user is on the navigation/tracking screen Android requires this base permission for any foreground service.
FOREGROUND_SERVICE_LOCATION Declared (highlighted) Geolocator position stream during active errand + Mapbox navigation service Android 14+ requirement: foreground services that access location MUST declare this type-specific permission.
ACCESS_BACKGROUND_LOCATION Removed via tools:node — Explicitly stripped from merged dependencies. App does NOT track in background.
ℹ️

Play Store declaration requirement

When submitting to Google Play, apps using FOREGROUND_SERVICE_LOCATION must complete the Prominent Disclosure form in Play Console, explaining in plain language that location is used during active errand delivery and turn-by-turn navigation. The form should reference both the TrackingService (runner tracking) and Mapbox Navigation (turn-by-turn) as the two tasks that need it.

05 — Summary

Putting it all together

🎯

The single sentence for reviewers

Lotto Runners uses FOREGROUND_SERVICE_LOCATION to power two runner-facing features that only activate on explicit user action: (1) live GPS location streaming to a Supabase database so customers can watch their runner move on a real-time map, and (2) Mapbox turn-by-turn navigation which internally runs a foreground service with foregroundServiceType="location" to keep GPS alive while the runner drives to pickup/dropoff addresses — all while the app is visibly open and in use. Background location is explicitly disabled.
